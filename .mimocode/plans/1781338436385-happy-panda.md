# Fix Railway Deploy — "Application failed to respond"

## Root Cause
Railway sets a `PORT` env var dynamically. The Dockerfile hardcodes `--port 8000`, so the app listens on the wrong port.

## Fix 1: Dockerfile — use $PORT
**File:** `src/app/skillpath-backend/Dockerfile`

Change CMD to read PORT from env:
```dockerfile
CMD ["sh", "-c", "uvicorn main:app --host 0.0.0.0 --port ${PORT:-8000}"]
```

## Fix 2: main.py — graceful startup error handling
**File:** `src/app/skillpath-backend/main.py`

The `create_tables()` in lifespan crashes the app if DATABASE_URL is unreachable. Wrap in try/except so the app at least starts and responds to /health.

## Fix 3: database.py — don't crash at import time
**File:** `src/app/skillpath-backend/src/db/database.py`

Currently `engine = create_async_engine(settings.DATABASE_URL)` runs at import time. If DATABASE_URL is wrong, import fails → app never starts. Make it lazy or add error handling.

## Files to modify
1. `src/app/skillpath-backend/Dockerfile` — PORT env var
2. `src/app/skillpath-backend/main.py` — graceful lifespan
3. `src/app/skillpath-backend/src\db\database.py` — safe engine init

## Verification
1. `docker build -t skillpath-backend src/app/skillpath-backend/` — builds OK
2. `docker run -e DATABASE_URL=... -e PORT=8000 skillpath-backend` — starts OK
3. Redeploy on Railway → `/health` returns `{"status": "ok"}`
